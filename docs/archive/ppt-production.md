## 一、定位与受众

| 项目 | 说明 |
|------|------|
| **受众** | 课程设计答辩老师 / 学期项目评审 |
| **场景** | 课堂答辩，限时 8–10 分钟讲解 |

共产出 **3 份阶段汇报（Week 1/2/3）+ 1 份最终总览答辩稿**，每份阶段 PPT 只呈现到该 Week 为止已完成的内容。

## 二、制作参数与设计规范

| 参数 | 选定值 |
|------|--------|
| **布局模板** | 学术答辩风格（深蓝标题栏 + 白底 + 红色装饰条）；`ppt-master` 模板 `academic_defense` |
| **画布格式** | PPT 16:9（1280×720） |
| **输出方式** | `python-pptx` 直接生成原生可编辑 PPTX |
| **内容形式** | 阶段汇报纯文字；最终稿预留图片占位框 |

设计规范：

- **主色调**：深蓝 `#003366`、红色装饰 `#CC0000`、强调蓝 `#0066CC`
- **背景**：白色 `#FFFFFF`
- **字体**：中文微软雅黑，英文/数字 Arial
- **字号**：封面 42pt，页面标题 20pt，正文 14–16pt
- **布局**：标题栏 + 内容区 + 页脚页码；左文右图或上图下文
- **图片占位**：浅蓝背景 + 蓝色边框矩形，标注「此处插入：xxx 截图」
- **质量基线**：中文编码正确（PPTX 内部 XML 无乱码）、每页底部页码「当前/总页数」、每页 bullet ≤ 5 条且每条 ≤ 2 行

## 三、4 份 PPT 结构与关键数据

### Week 1 阶段汇报（10 页，完成度约 25%）

封面 → 汇报提纲（6 项）→ 项目背景与目标 → Week 1 概览 → 技术选型与架构 → 数据库设计（12 张核心表）→ 用户模块（Session 认证）→ 资源模块（23 条资源）→ 后续计划 → 致谢

- **KPI**：11 提交 / 32 文件 / 12 表 / 23 资源
- 技术选型：Flask + MySQL + 原生 HTML/CSS/JS

### Week 2 阶段汇报（12 页，完成度约 75%）

封面 → 提纲（8 项）→ Week 1 回顾 → Week 2 概览 → 题库与练习 → 智能学习计划 → 学习进度统计 → 弱项识别与推荐 → 解析与答疑 → 当前完成度 → 后续计划 → 致谢

- **KPI**：200 题 / 6 模块 / 10 蓝图 / 75%
- 题型分布：单选 152 / 判断 34 / 多选 14
- 计划算法：科目权重 × 难度 + 弱项加权
- 弱项规则：答题量 ≥ 5 且正确率 < 60%

### Week 3 答辩（14 页，完成度 100%）

封面 → 提纲（8 项）→ 项目背景与目标 → Week 1 回顾 → Week 2 回顾 → Week 3 概览 → UI/UX 统一与动画 → 测试与演示数据 → 文档整理与归档 → 最终成果（7 模块全完成）→ 核心数据一览 → 后续计划与不足 → 系统演示信息 → 致谢 / Q&A

- **KPI**：42 提交 / 109 文件 / 18+ 端点 / 100%
- 演示信息：`localhost:8080` / `root` / `123456`

### 最终总览答辩稿（18 页，正式答辩稿）

封面 → 目录（12 项，双列）→ 项目背景与目标 → 技术选型与架构 → 数据库设计 → 用户与资源模块 → Week 1 小结 → 题库与练习 → 智能学习计划算法 → 进度统计与可视化 → 弱项识别与个性化推荐 → 题目解析与答疑 → Week 2 小结 → UI/UX 统一与动画交互 → 测试、Review 与演示数据准备 → 文档整理与项目归档 → 最终成果与核心数据 → 致谢 / Q&A

最终稿在对应页面预留图片占位框：

- 技术选型页 → 架构图
- 数据库设计页 → ER 图
- 用户与资源页 → 登录页/资源库截图
- 题库 / 计划 / 统计 / 推荐 / 答疑各页 → 对应模块截图
- UI/UX 页 → Dashboard 截图
- 最终成果页 → 底部 4 张截图缩略图

## 五、调研方法（素材如何收集）

阶段 PPT 文案由 3 个调研 agent 并行产出（Week 1/2/3 各一），最终汇总为答辩稿。每个 agent 先通读全局文档（`PRD`、`CONTEXT.md`、`docs/PROJECT_STATUS.md`、`docs/DEVELOPMENT_TIMELINE.md`、`docs/ROADMAP.md`、`AGENTS.md`、`README.md`）建立整体认知，再按 Week 调阅对应源码与文档，输出到 `.scratch/ppt-research/week<N>/`（`SUMMARY.md` / `KEY_POINTS.md` / `QUOTES.md` / `SCREENSHOTS.md`）。

各 Week 调研侧重：

- **Week 1（基础设施 + 核心数据）**：技术选型理由、12 张表设计、用户模块 Session/scrypt、23 条资源。源码侧重 `backend/app.py`、`db.py`、`routes/auth.py`、`routes/resources.py`、`frontend/login.html`、`js/api.js`。
- **Week 2（核心业务逻辑）**：题库筛选与多选比对、计划算法（权重×难度 + 弱项加权 50% + 基础/强化/冲刺三阶段）、进度统计、推荐规则、答疑。源码侧重 `routes/questions.py`、`answers.py`、`plans.py`、`progress.py`、`recommendations.py`、`comments.py` 及对应前端页。
- **Week 3（完善、Review、演示）**：UI/UX 统一与设计系统落地、动画交互（FLIP/热力图/加载 veil）、冒烟测试与 bug 修复、200 题扩容与 `resource_id` 关联、文档整理。源码侧重 `frontend/assets/styles.css`、`js/ui.js`/`shell.js`/`motion.js`、`backend/test_api.ps1`、`git log`。

**调研铁律**：所有数字（提交次数、文件数、题目数）必须能从 `git log` / `init_db.sql` / 文档二次核实，不编造；代码引用标注文件路径 + 行号；文案口语化，突出「从 0 到 1 的开发过程」而非产品手册。
