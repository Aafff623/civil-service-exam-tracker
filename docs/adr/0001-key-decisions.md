# ADR-0001:项目关键技术与流程决策

- **状态**:Accepted
- **日期**:2026-06-23(项目立项)
- **背景**:数据结构课程设计,面向大一、大二学生,要求简单、可演示、可答辩

## 决策

| 决策项 | 选择 | 理由 |
|---|---|---|
| 数据库 | MySQL 8.0+(PyMySQL) | 课程要求关系型;答辩机普遍预装;换机用 SQL 种子(`db/seed/`)复现,不依赖本机数据文件 |
| 推荐算法 | 规则驱动(弱项正确率 + 科目权重) | 非 ML;数据规模小,规则即可覆盖学习闭环价值,演示成本低 |
| 后端端口 | 5001 | 本地 5000 被占用 |
| 前端 | 原生 HTML/CSS/JS(无框架) | 降低答辩环境搭建成本,`python -m http.server` 即跑 |
| Mock fallback | 前端 JS 层 mock(`mock-api.js`) | 答辩机无 MySQL/Flask 时仍能演示全部页面与核心交互 |
| 密码哈希 | scrypt | 不存明文;课设安全需求下足够 |
| Issue tracker | 本地 `.scratch/<feature>/` markdown | 课设规模无需 GitHub Issues |

## 备选(为何没选)

- **SQLite**:课程要求 MySQL;且换机复现用 SQL 种子比 SQLite 文件更稳
- **ML 推荐**:课设数据不足,规则已能演示完整价值,引入 ML 反增复杂度
- **Vue / React**:增加答辩机搭建成本(构建工具 / Node),原生 JS 双击即跑更适合答辩场景

## 影响

这些决策决定了项目的部署形态(双终端 + Mock fallback)、数据复现方式(SQL 种子)、以及「简单可演示」的整体取向。后续若推翻其中任一项(如改用 ML 推荐、引入前端框架),应在本目录新建后续 ADR 说明动机与迁移代价。
